Traditional websites vs SPA

  1. Speed
  2. Code complexity
  3. SEO and social media sharing

Speed

Traditional website

  • Fast first load
  • Reload of page on every call

SPA

  • Slow first load due big js bundle
  • No need for reload

Code complexity

Traditional website

  • Frontend is highly coupled with backend
  • Hard to maintain state

SPA

  • SPA is similar to mobile apps. It's completely separate from backend
  • Great developer experience

SEO and social media sharing

Traditional website

  • Works without js(pure html/css from server)
  • Good SEO support

SPA

  • Need js to work properly
  • It takes additional effort to make it work with SEO and social media sharing

Component lifecycle

Lets write Clock

Constructor

Is equal to

Constructor

  • Don't forget to call super(props);
  • Good for init of state
  • Bad for using this.setState()
  • Bad for running side effects
    • No api calls
    • No dom manipulations

Render

Render

  • It's only required method in a class component.
  • Called only on
    • New props from parent
    • this.setState()
    • this.forceUpdate()
  • Should be pure
    • No side effects
    • No dom manipulations

componentDidMount

componentDidMount

  • Called after first render
  • You can call this.setState() here
  • Should be used for running side effects
    • Fetching data from server
    • Subscriptions
    • Timers
    • Manipulations that require dom

componentWillUnmount

componentWillUnmount

  • Called before component will be removed from DOM and destroyed
  • You should not call this.setState() here
  • Should be used for cancelling
    • Async requests to server
    • Subscriptions
    • Timers

componentWillUnmount

componentDidUpdate

  • Called after every update
  • Isn't called after initial render
  • Can be used for running side effects but don't forget to add condition

Component lifecycle. Let's extend

getDerivedStateFromProps

getDerivedStateFromProps

  • Called before each render
  • It's static!
  • You probably don't need it

shouldComponentUpdate

shouldComponentUpdate

  • Called before rendering when new props or state are being received.
  • By default return true
  • Should be used for performance optimization
  • Pure component

getSnapshotBeforeUpdate

getSnapshotBeforeUpdate

  • Called after render method, but before changes applied to DOM
  • Result will be passed to componentDidMount() as third argument
  • Help to save some information from the DOM (e.g. scroll position) before it is potentially changed

Routing

Two main ways

Hash router

site.com/#/users/1

  • Hash in the url
  • The server never sees the hash
  • Handled 100% in-browser
  • Do not by default result in page reload
  • Buttons and links are plain JS-free

Browser router

site.com/users/1

  • “Normal” URLs
  • The HTML5 History API
  • Multiple routes indexed by Google
  • Best choice by default
